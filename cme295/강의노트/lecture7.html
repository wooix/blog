01 — Overview

개요: LLM에게 오픈북 시험을 허락하다 Why give an LLM an open book?

시험에는 두 종류가 있습니다. 책을 덮고 순전히 기억으로 푸는 암기 시험과, 참고서와 계산기를 옆에 두고 푸는 오픈북 시험입니다.

결론부터 말하면, 지금까지 다룬 LLM(Large Language Model, 거대 언어 모델)은 전형적인 암기 시험 응시자였습니다. 사전학습(pretraining) 때 외운 지식만으로 답을 내놓았고, 시험이 끝난 뒤 세상이 바뀌어도 알 도리가 없었습니다. 이번 7강은 LLM에게 오픈북 시험을 허락하는 세 가지 방법을 다룹니다. 필요한 사실은 참고서에서 찾아보고(검색 증강 생성, Retrieval-Augmented Generation, RAG), 계산이나 실제 작업은 전용 도구에 맡기고(함수 호출, Function Calling), 문제가 풀릴 때까지 관찰·계획·행동을 스스로 반복하는(에이전트, Agent) 능력입니다.

원본 지난 시간과의 연결
지난 6강에서는 추론(reasoning), 즉 LLM이 답하기 전에 스스로 사고 과정을 길게 펼치는 능력을 다뤘습니다. 슬라이드는 LLM의 남은 약점을 이렇게 정리합니다: "지식은 고정되어 있고(knowledge is static), 실제 행동을 할 수 없다(cannot perform actions)." 이번 강의는 바로 이 두 가지 약점을 메우는 이야기입니다.
📖

검색 증강 생성(RAG)

모르는 사실은 외부 문서에서 찾아 프롬프트에 끼워 넣습니다. 참고서를 펼치는 것과 같습니다.

🛠️

함수 호출

계산·검색·메일 발송 같은 실제 작업은 전용 함수(도구)에 맡깁니다. 계산기를 손에 쥐는 것과 같습니다.

🔁

에이전트 & ReAct

관찰하고, 계획하고, 행동하는 루프를 답이 나올 때까지 반복합니다. 시험 문제를 풀며 앞뒤 페이지를 오가는 것과 같습니다.

02 — Motivation

왜 오픈북이 필요한가 The limits of a closed-book LLM

아무리 똑똑한 학생이라도 암기만으로는 한계가 있습니다. LLM도 마찬가지입니다.

🗓️

지식이 멈춰 있습니다

LLM의 지식은 사전학습 데이터가 수집된 시점, 즉 지식 마감일(knowledge cutoff)에 고정됩니다. 외부 2025년 11월 9일 촬영된 GPT-5 모델 카드 화면에는 지식 마감일이 2024년 9월 30일로 적혀 있었습니다 — 강의 시점 기준으로도 1년 이상 지난 정보입니다. OpenAI, GPT-5 모델 카드 ↗

📏

맥락 창이 유한합니다

아무리 맥락 창(context window)이 커져도(같은 모델 카드 기준 40만 토큰) 회사 내부 문서 전체나 실시간 데이터를 통째로 넣을 수는 없습니다.

🌀

정보가 많을수록 산만해집니다

외부 "Needle in a Haystack" 테스트에 따르면, 맥락이 길어질수록, 특히 찾아야 할 사실이 문서 중간 깊숙이 있을수록 검색 정확도가 떨어졌습니다. Kamradt, 2023, GitHub ↗

💳

토큰 하나하나에 값이 매겨집니다

API 요금은 입력·출력 토큰 수에 비례합니다. 맥락에 아무 문서나 욱여넣는 것은 곧 비용 낭비입니다. 외부 2025년 11월 10일 기준 OpenAI 가격표, 참고용(변동 가능) OpenAI Pricing ↗

결국 필요한 것은 "모든 걸 외우기"가 아니라 "필요할 때 정확한 페이지만 펼쳐 보기"입니다. 이것이 RAG의 출발점입니다.

03 — RAG

RAG란 무엇인가: 참고서를 옆에 두고 답하기 Retrieval-Augmented Generation

RAG(Retrieval-Augmented Generation, 검색 증강 생성)는 이름 그대로 검색(Retrieval)한 정보로 생성(Generation)을 증강(Augmentation)하는 방법입니다. 원본 Lewis et al., 2020 ↗

원본 핵심 아이디어
슬라이드의 정의는 간결합니다: "프롬프트를 관련 있는 정보 조각으로 증강한다(augment prompt with relevant pieces of information)." 지식을 파라미터 안에 욱여넣는 대신, 필요할 때마다 외부 저장소 𝒟(문서 모음)를 뒤져 관련 조각을 찾고, 이를 질문 앞에 덧붙여 LLM에게 건네줍니다.

세 단계: 검색 → 증강 → 생성

①

검색(Retrieve)

사용자 질문과 지식 베이스 사이의 유사도 연산으로 관련 문서를 찾습니다. 참고서 색인에서 관련 페이지를 찾는 일과 같습니다.

②

증강(Augment)

찾아낸 정보를 사용자 프롬프트 앞에 붙입니다. 책에서 찾은 페이지를 시험지 옆에 펼쳐두는 것과 같습니다.

③

생성(Generate)

증강된 프롬프트를 LLM에 넣어 응답을 만듭니다. 펼쳐둔 페이지를 참고해 답안을 쓰는 것과 같습니다.

지식 베이스는 어떻게 만드나요?

검색을 하려면 먼저 뒤질 대상이 있어야 합니다. 원본 슬라이드는 지식 베이스 구축을 세 단계로 그립니다.

📚

수집(Collect)

문서 1, 문서 2, … 원본 자료를 모읍니다.

✂️

분할(Divide)

청킹(Chunking, 문서 분할)으로 각 문서를 검색하기 좋은 크기의 조각(청크, chunk)으로 나눕니다.

🧬

임베딩(Embed)

각 청크를 숫자 벡터로 바꿉니다. 예: [0.48, 0.33, …, -0.51]

하이퍼파라미터: 임베딩 크기, 청크 크기, 청크 간 겹침(overlap) — 슬라이드에서 직접 언급된 조정 대상입니다.

04 — Retrieval

검색은 왜 2단계로 나뉘나요? Candidate retrieval + Ranking

도서관에서 책을 찾을 때, 우리는 먼저 서가를 넓게 훑어 후보를 추린 다음(재현율 중시), 그중 몇 권만 실제로 펼쳐 읽어보며 진짜 필요한 책을 고릅니다(정밀도 중시). 검색도 똑같은 2단계 구조를 씁니다.

1️⃣

1단계 — 후보 검색(Candidate Retrieval)

목표는 재현율(Recall) 극대화입니다. 정답이 될 만한 후보를 넓게, 빠르게 모읍니다.

2️⃣

2단계 — 재순위화(Ranking)

목표는 정밀도(Precision) 극대화입니다. 추려진 소수의 후보만 정밀하게 다시 채점합니다.

검색 방식 비교해보기

1단계 후보 검색에는 크게 세 방식이 쓰입니다. 탭을 눌러 차이를 살펴보세요.

의미 기반 검색 — 바이 인코더(Bi-encoder)

외부 질의와 청크를 각각 따로 인코더에 통과시켜 벡터로 만든 뒤, 두 벡터의 유사도를 잽니다. 이런 구조를 바이 인코더("bi-encoder")라 부르며, 대표적으로 Sentence-BERT가 있습니다. Reimers & Gurevych, 2019 ↗

도서관 색인에서 제목만 훑어보며 후보를 고르는 것과 같습니다 — 질의와 문서를 미리 따로따로 벡터로 바꿔두면(색인화), 새 질문이 올 때마다 저장된 벡터와 비교만 하면 되니 지식 베이스 전체를 대상으로도 매우 빠르게 훑을 수 있습니다. 다만 표현이 비슷하면 실제 내용이 달라도 벡터가 가까워지는 함정이 있습니다.

키워드 기반 검색 — BM25

원본 BM25는 질의 단어가 문서에 얼마나(그리고 얼마나 희귀하게) 등장하는지를 세어 점수를 매기는 고전적인 키워드 매칭 알고리즘입니다.

색인에서 정확한 단어를 검색창에 입력하는 것과 같습니다 — 의미는 몰라도 정확한 이름·약어·숫자 일치는 절대 놓치지 않습니다. 다만 동의어나 문맥은 이해하지 못합니다.

하이브리드 검색(Hybrid Search)

원본 의미 기반 검색과 BM25 점수를 함께 조합해 최종 후보를 뽑습니다. 색인을 훑는 눈과 검색창을 두드리는 손을 동시에 쓰는 셈입니다 — 서로의 약점을 보완합니다.

직접 확인해보기: 왜 "Cuddly는 어디 있나요?"가 까다로운가

원본 슬라이드는 재미있는 예시를 듭니다. 곰인형 "Cuddly"를 찾는 질문 "Where is Cuddly?"를 던졌을 때, 검색 방식에 따라 완전히 다른 문장이 상위로 올라옵니다. 아래에서 질의와 검색 방식을 바꿔가며 순위가 어떻게 뒤집히는지 직접 확인해보세요.

종합·해석 이 데모에 대하여
문장 4개와 "어떤 방식이 어떤 문장을 고르는가"라는 결론은 강의 원본 예시를 그대로 재현했습니다. 다만 코사인 유사도 계산에 쓰인 벡터 값 자체는 이해를 돕기 위해 직접 구성한 3차원 예시입니다(실제 Sentence-BERT 임베딩은 수백 차원). 아래 막대 길이와 점수는 이 벡터로부터 브라우저가 실시간으로 재계산한 값입니다.

05 — Advanced RAG

고급 RAG 기법: 파이프라인 곳곳의 병목 풀기 Advanced RAG techniques

오픈북 시험도 참고서를 잘 쓰는 사람과 서투르게 쓰는 사람의 점수 차가 큽니다. RAG 파이프라인의 5개 단계 각각에는 나름의 병목이 있고, 고급 RAG 기법들은 그 병목을 하나씩 풀어줍니다. 아래 스텝을 눌러가며 단계별 문제와 해법을 따라가 보세요.

검색 품질을 재는 4가지 자

오픈북 시험에서 "참고서를 잘 썼는지"를 두 질문으로 나눠 볼 수 있습니다 — 필요한 페이지를 빠짐없이 찾았는가(재현율)와 찾은 페이지가 전부 쓸모 있었는가(정밀도). 원본 슬라이드는 이를 정량화하는 4가지 지표를 제시합니다.

정규화 누적 이득(NDCG@k)

NDCG(Normalized Discounted Cumulative Gain at k)는 관련 있는 문서가 상위 k개 중 얼마나 위쪽에 몰려 있는지를 잽니다. 같은 정답이라도 1등에 있으면 만점에 가깝고, k번째 언저리에 있으면 점수가 깎입니다.

DCG@k = Σi=1k reli / log₂(i+1),   reli ∈ {0, 1}
NDCG@k = DCG@k / IDCG@k  (IDCG@k = 이상적인 순서로 정렬했을 때의 DCG@k)

역순위(RR@k)

RR(Reciprocal Rank, 역순위)는 "첫 정답을 만나기까지 몇 번 뒤졌는가"를 잽니다. 색인에서 원하는 페이지를 단번에 찾으면 1점, 열 번째 만에 찾으면 0.1점입니다.

RR = 1 / rank  (rank = 첫 번째 관련 청크의 순위)

재현율(Recall@k)과 정밀도(Precision@k)

재현율(Recall)은 "존재하는 정답을 얼마나 빠짐없이 찾았는가", 정밀도(Precision)는 "가져온 것 중 얼마나 진짜 정답인가"를 잽니다. 1단계 후보 검색은 재현율을, 2단계 재순위화는 정밀도를 각각 겨냥합니다.

Recall@k = |상위 k개 중 관련 문서| / |전체 관련 문서|
Precision@k = |상위 k개 중 관련 문서| / k
06 — Tool calling

함수 호출: LLM에게 계산기를 쥐여주다 Function (Tool) Calling

오픈북 시험에서 참고서만큼 요긴한 것이 계산기입니다. RAG가 "찾아보기"를 담당한다면, 함수 호출(Function Calling, 흔히 Tool Calling·도구 호출로도 부름)은 "실제로 해보기"를 담당합니다.

"Tool calling […] allows autonomous systems to complete complex tasks by dynamically accessing and [may act] upon external resources."— 외부 IBM, "Tool calling overview"

예시로 보기: 근처 곰인형 찾기

원본 "Find a bear near me!"라는 요청을 받은 일반 LLM은 "죄송해요, 근처에 어떤 곰이 있는지 몰라요"라고 답할 수밖에 없습니다. 위치 정보에 접근할 방법이 없기 때문입니다. 슬라이드는 이를 find_teddy_bear.py라는 함수로 해결합니다 — GPS 좌표를 받아 외부 API를 호출하고, 가장 가까운 곰인형의 이름·거리·기분을 담은 구조화된 결과를 돌려주는, 설명이 잘 달린(docstring) 파이썬 함수입니다.

동작 원리: 세 걸음

1️⃣

인자 찾기

LLM이 함수 API 명세와 사용자 요청을 보고, 어떤 함수를 어떤 인자로 불러야 할지 스스로 판단합니다. find_teddy_bear(location=(37.42,-122.17))

2️⃣

실제로 호출

백엔드가 그 함수를 실제로 실행하고 결과(JSON 등)를 돌려받습니다. {"name":"Teddy", …}

3️⃣

결론 도출

LLM이 함수 결과를 읽고 사람이 이해할 자연어 응답을 만듭니다.

모델에게 도구 쓰는 법을 어떻게 가르치나요?

🎓

방법 1 — 학습(Training)

원본 "함수 API + 사용자 요청" → "정답 함수 호출"이라는 대화 기록 쌍으로 모델을 직접 파인튜닝합니다. 외부 자기지도 방식으로 언제·어떻게 도구를 쓸지 스스로 학습하게 만든 초기 사례로 Toolformer가 있습니다. Schick et al., 2023 ↗

💬

방법 2 — 프롬프팅(Prompting)

원본 함수 API에 자세한 사용 설명을 덧붙여 프롬프트로 그냥 건네줍니다. "이런 설명은 누가 쓰나요?" — 슬라이드의 답: 사람이 쓴 정답 예시(SFT pair)를 평가 기준 삼아, 강력한 추론 모델(reasoning model)에게 설명 자체를 작성시키는 방법도 있습니다.

도구가 많아지면: 선택과 표준화

실전에서는 곰인형 찾기·안아주기·선물 보내기·메시지 전송처럼 도구가 수십 개로 불어납니다. 모든 도구 설명을 매번 프롬프트에 다 넣으면 맥락이 넘치고 성능도 떨어집니다. 원본 슬라이드는 두 가지 해법을 제시합니다.

외부 도구 라우팅(Tool Selection)
먼저 가벼운 라우터가 이번 요청에 관련 있을 법한 도구만 추려내고, 그 다음에야 LLM이 추려진 목록만 보고 최종 호출을 결정합니다. Robert et al., 2024 ↗
원본 표준화: MCP(Model Context Protocol)
모든 LLM이 모든 도구와 각자 다른 방식으로 연결되면(LLM × 도구 조합마다 중복 구현) 비효율적입니다. MCP(Model Context Protocol, 모델 컨텍스트 프로토콜)는 "LLM ↔ 도구·데이터"를 잇는 표준 규격입니다. MCP 호스트 안의 클라이언트가 MCP 서버에 접속하면, 서버가 노출한 도구(Tools)·프롬프트(Prompts)·자원(Resources)을 통일된 방식으로 쓸 수 있습니다. 외부 Anthropic, 2024 ↗

비유하자면 MCP는 사내 직원(LLM)과 여러 부서의 도구 창고를 잇는 표준 사내 시스템과 같습니다. 콘센트 규격이 통일되면 어떤 가전이든 꽂기만 하면 되는 것과 비슷합니다.

07 — Agents

에이전트란 무엇인가: 답이 나올 때까지 스스로 반복하다 Agents & the ReAct loop

"An agent is a system that autonomously pursues goals and completes tasks on a user's behalf."— 원본 강의 슬라이드 정의

참고서(RAG)와 계산기(함수 호출)를 손에 쥐여줬다고 시험이 저절로 끝나지는 않습니다. 문제를 풀려면 "뭘 알아야 하지? → 뭘 해야 하지? → 해보자 → 다시 확인"을 답이 나올 때까지 반복해야 합니다. 이 반복 전략을 갖춘 시스템이 에이전트(Agent)입니다.

세 가지 응답 방식 비교

방식흐름비유
전통적 방식질문 → LLM → 답변즉석 구술 시험
추론(6강)질문 → LLM → (내부) 추론 → 답변암산으로 오래 생각한 뒤 답하기
에이전트(7강)질문 → LLM → 도구 호출 → LLM → … → 답변참고서·계산기를 오가며 답을 완성하기

ReAct: 관찰하고, 계획하고, 행동하라 ReAct = Reason + Act

원본 ReAct("Re"ason + "Act")는 에이전트가 이 반복을 구조화하는 대표적인 프레임워크입니다. Yao et al., 2022 ↗ 매 사이클마다 세 단계를 거칩니다.

👀

관찰(Observe)

지금까지의 행동 결과와 스스로 아는 것을 종합해, 현재 무엇을 알고 무엇을 모르는지 명시적으로 정리합니다.

🗺️

계획(Plan)

어떤 작업이 필요하고 어떤 도구를 호출해야 하는지 구체적으로 정합니다.

⚙️

행동(Act)

API를 호출하거나 문서 데이터베이스를 뒤지는 등, 계획을 실제로 실행합니다.

이 관찰 → 계획 → 행동 루프는 만족스러운 답이 나올 때까지 여러 번 돌 수 있습니다. 아래에서 슬라이드의 "테디베어가 추워요" 예시를 직접 단계별로 눌러 따라가 보세요.

종합·해석 "에이전틱" 관점
루프를 한 걸음 물러서서 보면, 입력(사용자 요청)이 들어가 출력(응답)이 나오는 하나의 검은 상자처럼 보입니다 — 슬라이드는 이를 "온도조절기 에이전트(Thermostat agent)"라 부릅니다. 내부에서 몇 번을 관찰·계획·행동했는지는 감춰진 채, 겉으로는 하나의 지능적인 행위자처럼 동작합니다.

에이전트가 여럿이라면? Multi-agent systems & A2A

원본 온도조절기 에이전트 하나만 있는 게 아닙니다. 에너지 관리 에이전트, 재실(在室) 감지 에이전트, 공기질 에이전트처럼 각자의 전문 영역을 가진 에이전트가 여러 개 존재할 수 있습니다. 문제는 — 이들이 서로 어떻게 말을 걸 수 있을까요?

원본 표준화: A2A(Agent2Agent Protocol)
A2A(Agent2Agent, 에이전트 간 프로토콜)는 서로 다른 에이전트끼리 대화하기 위한 표준입니다. 각 에이전트는 AgentCard(이름·주소·버전·보유 기술 목록)로 자신을 소개하고, AgentSkill로 할 수 있는 일을 설명하며, AgentExecutor가 실제 실행을 맡습니다. 외부 Google, 2025 ↗

MCP가 "직원(에이전트)과 도구 창고를 잇는 사내 시스템"이라면, A2A는 "여러 부서 직원들끼리 업무를 주고받는 사내 메신저"에 비유할 수 있습니다 — 하나는 수직(에이전트→도구), 다른 하나는 수평(에이전트↔에이전트) 연결입니다.

08 — Safety & closing

에이전트에게 손발을 줬다면, 안전장치도 함께 Safety and closing thoughts

계산기를 쥐여준 학생이 시험 중 옆자리 답안을 몰래 촬영해 전송한다면 어떨까요? 실제 세계에 작용할 수 있는 에이전트에게는 이런 위험이 진짜로 존재합니다.

마무리: 오늘 배운 것과 남은 과제

원본 슬라이드가 던지는 다섯 가지 정리입니다.

  • 환각(Hallucination)은 여전히 (큰) 문제입니다 — 근거를 줘도 LLM이 이를 무시하거나 지어낼 수 있습니다.
  • 추론 능력이 병목입니다 — 파인튜닝이 도움이 되지만 어렵고, 새로운 추론 능력은 언제나 반갑습니다.
  • 평가가 어렵습니다 — "잘 답했다"를 자동으로 채점하기가 쉽지 않습니다(다음 강의 주제).
  • 처음엔 단순하게 시작해 점진적으로 규모를 키우는 편이 좋습니다.
  • 처음엔 강력한 모델로 시작하고, 크기 최적화는 나중에 하는 편이 좋습니다.
  • 투명성/관측 가능성(transparency/observability)이 사용자 신뢰와 디버깅에 도움이 됩니다.
원본 보너스: 일상 속 AI 에이전트
슬라이드는 개인적으로 가장 좋아하는 활용 사례로 코딩 에이전트를 꼽습니다 — 코드 편집기 옆 패널에서 find_teddy_bear.py 같은 파일의 정확성을 스스로 점검하고, API 응답 검증 누락이나 타입 힌트 호환성 같은 문제를 짚어주는 모습입니다.
