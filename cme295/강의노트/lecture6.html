01 — Overview

추론이란 무엇인가? What is "reasoning"?

학원에서 모의고사를 채점하는 두 가지 방식을 떠올려 봅시다.

한 선생님은 정답지를 놓고 "80점 이상이면 합격"처럼 절대 기준으로 채점합니다. 다른 선생님은 같은 반 학생 여러 명이 같은 문제를 풀게 한 뒤, 그 반 안에서 상대적으로 잘 푼 답안에만 칭찬 스티커를 주는 상대평가를 씁니다. 이번 6강에서 다루는 추론 모델(Reasoning Model)의 핵심 학습법인 GRPO(Group Relative Policy Optimization, 그룹 상대 정책 최적화)는 두 번째 선생님과 닮았습니다 — 정답을 절대적으로 채점해 줄 별도의 "가치 모델"(Value Model) 없이도, 같은 문제를 여러 번 풀어본 답안끼리 비교해 상대적으로 잘한 풀이를 강화하는 방식으로 모델이 스스로 생각하는 법을 익히게 만드는 것이지요. 이 비유는 앞으로 GRPO·테스트 시점 스케일링·증류를 설명할 때마다 다시 불러오겠습니다.

원본 잠정적 정의
슬라이드는 추론(Reasoning)을 "문제를 풀어내는(solve) 능력"으로 잠정 정의합니다. "Stanford Transformers & LLMs 강의 코스 코드가 뭐야?"처럼 그냥 알면 답하는 질문은 추론이 필요 없지만, "곰이 2020년에 태어났다. 지금 몇 살일까?"처럼 중간 계산이 필요한 질문은 추론이 필요합니다.

바닐라 LLM의 약점, 그리고 오늘의 초점

지난 5강까지 배운 사전학습(pretraining)·미세조정(finetuning)·선호도 튜닝(preference tuning)을 거친 "바닐라" LLM도 모방·아이디어 생성과 코드 작성엔 강하지만, 네 가지 약점이 있습니다: 제한된 추론력, 정적인 지식, 행동 불가, 평가의 어려움. 7·8강이 뒤의 세 가지를 다루고, 이번 6강은 제한된 추론력을 정면으로 다룹니다.

추론 모델은 언제부터, 어떻게 알아볼까?

원본 슬라이드가 제시한 타임라인에 따르면 OpenAI o1-preview(2024-09-12)를 시작으로 Gemini 2.0 Flash Thinking(2024-12-19), DeepSeek R1(2025-01-20), Grok 3 Beta(2025-02-19), Claude 3.7 Sonnet(2025-02-24), Magistral(2025-06-10) 순으로 주요 추론 모델이 쏟아져 나왔습니다. 추론 모델은 보통 UI에 "Thinking"·"Extended thinking" 표시와 함께, 최종 답 앞에 사고 요약(thought summary)을 보여줍니다 — 다만 전체 사고 사슬(chain of thought)은 대개 숨겨져 있습니다.

외부 과금 방식의 차이
2025년 11월 4일 기준 스크린샷을 보면(OpenAI pricing page, Anthropic developer guide, Google developer guide), OpenAI gpt-5는 입력 $1.25 / 출력 $10.00(100만 토큰당)이며 "사고 토큰은 API에 보이지 않아도 컨텍스트 창을 차지하고 출력으로 과금"됩니다. Anthropic 개발자 가이드는 "청구되는 것은 요약이 아니라 원 요청이 생성한 전체 사고 토큰"이라 명시하고, Google Gemini 2.5 Pro는 "사고 토큰 포함 출력가"를 별도로 표기합니다. 즉 추론 모델은 눈에 보이지 않는 사고 과정까지 값을 치르는 구조입니다.

추론을 어떻게 재는가: 벤치마크와 Pass@k

추론 능력을 측정하는 벤치마크는 크게 코딩(HumanEval, CodeForces, SWE-bench — 문제→코드→테스트케이스로 자동 검증)과 수학(AIME, GSM8K — 문제→풀이→정답과 대조)으로 나뉩니다. 둘 다 "정답 여부를 기계적으로 검증할 수 있다"는 공통점이 있는데, 이 성질이 뒤에서 GRPO의 보상 설계에 그대로 쓰입니다.

Pass@k = "적어도 k번 시도 중 하나가 성공할 확률"Chen et al., 2021, Evaluating Large Language Models Trained on Code

이 값을 그냥 n번 중 c번 성공했다고 c/n으로 추정하면 편차가 커서, 논문은 불편추정량(unbiased estimator)을 씁니다: Pass@k = 1 − C(n−c, k) / C(n, k) (n번 샘플링해서 c번 성공했을 때). 특수하게 k=1이면 Pass@1 = c/n입니다. 이 식이 "무엇을 측정하는지" 풀어보면, "k번 뽑을 때 뽑은 것이 전부 실패작(n−c개 중에서만 뽑을) 확률"을 1에서 뺀 것 — 즉 "적어도 하나는 성공"할 확률입니다. 원본 슬라이드는 여기에 Cons@k("Consensus@k", 다수결(majority voting) 답과 정답을 비교하는 지표, DeepSeek-AI 2025)도 함께 소개합니다. 아래 위젯에서 이 공식을 직접 재계산해 볼 수 있습니다 → 스케일링 섹션.

사고 없이 답변

원본 Few-shot 예시에 풀이 과정이 없으면, 모델도 풀이 없이 바로 답만 냅니다.

Q: How old is this bear? A: 4. Q: How old will the bear be next year? → LLM → A: 5.

중간 계산이 드러나지 않아, 값이 조금만 복잡해져도 틀리기 쉽습니다.

사고 사슬(Chain-of-Thought)과 함께

외부 Wei et al., 2022의 Chain-of-Thought Prompting Elicits Reasoning in Large Language Models ↗가 제안한 방식으로, few-shot 예시에 풀이 과정을 넣어주면 모델도 풀이 과정을 스스로 쓰게 됩니다.

Q: How old is this bear? A: The bear was born in 2020. It is therefore 4. Q: How old will the bear be next year? → LLM → A: It will be one year older than its age this year, which was 4. Hence, it will be 5.

추론 모델을 만드는 핵심 아이디어는 "CoT를 훨씬 더 큰 규모로 하자"는 것입니다 — 다음 섹션에서 이어집니다.

03 — Why RL

왜 강화학습으로 추론을 가르치나? RL with verifiable rewards

Idea: 답을 내놓기 전에 "생각"하도록 모델을 유인합니다(incentivize).

모의고사 채점 비유로 돌아가 봅시다. 학생에게 "풀이 과정을 꼭 쓰라"고 시키고 싶다면 두 가지 방법이 있습니다: (1) 선생님이 모범 풀이를 일일이 써준 뒤 학생이 그것을 베껴 외우게 하거나(SFT, Supervised Fine-Tuning — 정답 예시를 모방해 학습하는 단계), (2) "맞았는지 틀렸는지"만 알려주고 학생이 스스로 풀이 방식을 찾아내게 하는 것입니다. 원본 슬라이드는 세 가지 이유로 (2)번, 즉 RL을 택합니다: 사고 사슬을 사람이 직접 손으로 쓰는 것은 비현실적이고(SFT data by hand impractical), 사람이 쓴 방식에만 모델을 가두고 싶지 않으며, 무엇보다 "문제를 풀었는가?"라는 자연스러운 검증 가능 보상(natural verifiable reward)이 이미 있기 때문입니다.

1️⃣

보상 1 — 형식 확인

원본 응답이 <think> ... </think> ANSWER 템플릿을 지켰는지만 규칙으로 확인합니다. DeepSeek-R1 논문의 실제 샘플 응답을 보면 "Wait, wait. Wait. That's an aha moment I can flag here."처럼 스스로 되짚어보는 문장이 자연스럽게 등장합니다.

2️⃣

보상 2 — 정답 확인

원본 코드 문제는 테스트 케이스 실행으로, 수학 문제는 정답과의 대조로 자동 채점합니다. 사람 라벨러 없이도 대량으로 RL을 돌릴 수 있게 하는 핵심입니다.

원본 최종 보상
Reward = formatting(think 구분자 준수) + accuracy(정답 여부) — 이 단순한 두 항만으로 RL을 돌린 것이 DeepSeek-R1-Zero입니다.
외부 실제로 좋아졌나?
DeepSeek-R1(DeepSeek-AI, 2025) ↗ 논문이 보고한 R1-Zero의 AIME 2024 정확도는 학습 초반 pass@1 약 15%에서 약 8,000스텝 뒤 약 71%까지, 다수결 채점인 cons@16 기준으로는 약 86%까지 올라갑니다 — 별도의 정답 예시(SFT) 없이 오직 "맞았다/틀렸다" 보상만으로 이만큼 개선된 것입니다.
04 — GRPO

GRPO: 추론 RL에서 가장 널리 쓰이는 알고리즘 Group Relative Policy Optimization

GRPO = Group Relative Policy Optimization(그룹 상대 정책 최적화). 외부 DeepSeekMath(Shao et al., 2024) ↗가 제안했습니다.

5강에서 배운 PPO(Proximal Policy Optimization, 근접 정책 최적화)의 목적함수는 "어드밴티지(advantage)를 최대화하되, 옛 모델에서 너무 멀어지지 말라"는 두 항의 합이었습니다. GRPO도 같은 뼈대를 쓰지만, advantage를 계산하는 방식이 다릅니다 — 별도의 가치 모델(value model)로 절대적인 기준선을 추정하는 대신, 같은 질문 q에 대해 G개의 응답을 한 번에 뽑아 그 그룹 안에서 상대적으로 비교합니다. 모의고사 비유에서 "같은 반 학생들끼리 상대평가"하던 두 번째 선생님이 바로 이것입니다.

종합·해석 위 다이어그램은 슬라이드의 GRPO 파이프라인 그림을 단순화해 재구성한 것입니다.

Âi = ( ri − mean({r₁,…,r_G}) ) / std({r₁,…,r_G})그룹 내 보상을 평균 0, 표준편차 1로 표준화(z-score)한 값 — "그룹 평균보다 얼마나 잘/못했는가"만 남깁니다. 슬라이드 원문, Shao et al., 2024

직접 계산해보기: 그룹 상대 어드밴티지 계산기

같은 질문에 4개의 응답(G=4)을 뽑았다고 하고, 각 응답의 보상(0=완전히 틀림 ~ 1=완전히 맞음)을 슬라이더로 조절해 보세요. GRPO 공식을 그대로 재계산합니다.

PPO와 무엇이 같고 무엇이 다른가

같은 점

원본 두 알고리즘 모두 새 정책과 이전 정책의 확률비(ratio) ρ = π_θ/π_θold를 쓰고, 이를 [1−ε, 1+ε] 범위로 클리핑(clipping)해 정책이 한 번에 너무 크게 바뀌지 않도록 막습니다.

다른 점

원본 ① GRPO는 advantage 항에 KL 페널티를 별도 항으로 명시적으로 더합니다(PPO는 보상 자체에 KL을 섞기도 함). ② PPO는 응답마다 가치 모델(Value Model)과 GAE로 advantage를 추정하지만, GRPO는 그룹 통계만으로 advantage를 계산해 가치 모델이 아예 필요 없습니다 — 메모리·연산이 크게 절약됩니다.

부작용: 왜 응답이 점점 길어질까?

원본 R1-Zero를 학습시키며 관찰된 현상입니다 — RL 스텝이 진행될수록 평균 응답 길이가 계속 늘어납니다(약 500 토큰 → 약 10,000 토큰). 원인은 advantage 공식의 1/|o_i| 항(응답 토큰 수로 나누는 평균)에 있습니다: 정답(A>0)일 때는 짧을수록 토큰당 보상이 커서 짧은 정답을 더 강하게 밀어주지만, 오답(A<0)일 때는 반대로 긴 오답일수록 토큰당 페널티가 희석되어 덜 억제됩니다 — 즉 "오답이면 차라리 길게 써서 벌점을 나눠 받자"는 나쁜 유인(bad incentive)이 생깁니다.

DAPO의 해법

외부 DAPO(Yu et al., 2025) ↗는 응답별로 나누지 않고, 그룹 전체 토큰 수의 합으로 한 번에 나눠 모든 토큰이 손실에 동일하게 기여하도록 합니다. 비대칭 클리핑 범위 ε_low/ε_high로 다양성도 함께 유도합니다.

Dr. GRPO의 해법

외부 Understanding R1-Zero-Like Training(Liu et al., 2025) ↗는 아예 1/|o_i| 항을 제거해 토큰 수 편향 자체를 없앱니다("GRPO Done Right"). 같은 보상 수준에서 더 짧은 응답으로도 정답률을 유지한다고 보고합니다.

외부 Token-Budget-Aware LLM Reasoning(Han et al., 2024) ↗은 문제 난이도에 따라 토큰 예산을 동적으로 추정·할당해, 성능 손실을 최소화하며 사고 토큰 비용을 줄입니다.

예산 강제(Budget forcing)

외부 s1: Simple test-time scaling(Muennighoff et al., 2025) ↗은 사고 과정을 강제로 끊거나, "Wait" 문구를 덧붙여 더 길게 이어가게 만드는 단순한 개입만으로 test-time scaling 효과를 얻습니다.

외부 "연속(continuous)" 사고
Coconut(Hao et al., 2024) ↗은 사고 과정을 사람이 읽는 텍스트 토큰이 아니라, 모델의 마지막 은닉 상태(hidden state)를 그대로 다음 입력으로 되먹이는 "연속 잠재공간 사고"를 제안합니다 — 단어로 디코딩하지 않으니 여러 후보 경로를 동시에 품을 수 있습니다.
06 — R1 recipe

DeepSeek R1의 5단계 학습 레시피 Stitching it all together

「해결 →」 버튼으로 진행하며, 각 단계가 이전 단계의 한계를 어떻게 풀어내는지 따라가 보겠습니다.

원본 R1-Zero는 V3-Base(MoE, 총 671B 파라미터 중 토큰당 약 37B만 활성화)에 GRPO만 바로 적용한 "개념 증명(proof of concept)"이고, R1은 그 위에 여러 단계를 더 쌓은 "전체 파이프라인(full pipeline)"입니다.

07 — Distillation

증류: 큰 추론력을 작은 모델로 옮기기 And what about distillation?

잘 푸는 선배의 풀이 과정을 그대로 베껴 배우는 후배를 떠올려 봅시다.

2강에서 배운 지식 증류(Distillation)는 교사 모델과 학생 모델의 다음 토큰 확률분포를 손실함수로 직접 맞추는 방식이었습니다(soft label 매칭). 이번 강의의 증류는 다릅니다 — 원본 목표가 "추론 흔적(reasoning trace)을 SFT로 학습"하는 것으로 바뀝니다: R1이 정해둔 문제들(targeted x)에 대해 응답 전체를 생성하면, 그 전체 응답 텍스트로 학생 모델(예: Qwen, Llama)을 지도 미세조정(SFT)합니다 — 확률분포를 맞추는 게 아니라, 통째로 된 "모범 답안"을 베껴 쓰게 하는 셈입니다.

원본 "잘한" 컴퓨트 사용
DeepSeek-R1 논문은 같은 32B 모델에 ① RL을 처음부터 직접 돌린 경우(R1-Zero-Qwen-32B)와 ② 큰 모델(R1)의 추론 흔적으로 증류만 한 경우(R1-Distill-Qwen-32B)를 비교합니다. 결과는 증류 쪽이 크게 앞섭니다 — 작은 모델을 직접 RL로 단련하는 것보다, 더 큰 모델이 이미 찾아낸 추론 패턴을 베껴 배우는 쪽이 같은 컴퓨트로 더 나은 성능을 냅니다.
